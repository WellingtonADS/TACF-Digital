Plano de Projeto: TACF-Digital (HACO)

Este documento detalha o planejamento, arquitetura e requisitos para a implementação do sistema de agendamento do Teste de Avaliação de Condicionamento Físico (TACF) para o Hospital da Aeronáutica de Canoas.

1. Visão Geral do Projeto

1.1 Justificativa e Problema

O agendamento do TACF no HACO é historicamente realizado via planilhas manuais ou sistemas genéricos.

Problema: Falta de controle de concorrência (mais de 21 militares agendando simultaneamente o mesmo turno), exposição de dados nominais (ferindo a LGPD) e dificuldade na geração de listas de presença oficiais.

Solução: Uma plataforma web dedicada, com validação atômica em nível de banco de dados e isolamento lógico de informações.

1.2 Objetivos Estratégicos

Quórum Operacional: Garantir que cada turno tenha no mínimo 8 e no máximo 21 participantes.

Meta Institucional: Facilitar o agendamento para atingir a meta de 380 militares avaliados por semestre.

Privacidade: Implementar o isolamento de dados militares (Row Level Security).

Sazonalidade: Operar exclusivamente nas janelas de FEV-MAI e SET-NOV.

1.3 Patrocinadores e Stakeholders

Patrocinador: Comando do HACO.

Stakeholders: Coordenação do TACF, Seção de Preparo Físico, Oficiais de Pessoal e Militares da Guarnição.

2. Análise de Atores e Perfis de Acesso

2.1 Perfis

Militar (Avaliado): Acesso padrão. Pode preencher seu perfil, visualizar vagas e realizar/cancelar seu próprio agendamento.

Coordenador (Admin): Acesso total. Gere turnos, visualiza estatísticas de adesão, aprova trocas e gera relatórios para impressão.

Sistema (Automação): Gerencia a numeração de ordem e as regras de quórum automaticamente.

3. Estrutura Tecnológica (Stack)

Gerenciador de Pacotes: Yarn.

Linguagem: TypeScript (Strict Mode).

Framework: React 18 + Vite 5.

Estilização: Tailwind CSS v4 (Foco em Web Responsiva, sem PWA).

Backend: Supabase (PostgreSQL 15+).

Banco de Dados: PostgreSQL com funções RPC e Row Level Security (RLS).

Geração de Relatórios: jsPDF e CSS Print Media.

4. Arquitetura de Dados e Fluxos

4.1 Relacionamento de Entidades (ER)

Profiles: Armazena dados nominais (Nome, Posto, SARAM, Gênero). Chave estrangeira para auth.users.

Shifts (Turnos): Armazena data, período (Manhã/Tarde), capacidade máxima e lista de aplicadores.

Schedules (Agendamentos): Tabela de junção entre Profile e Shift. Contém o order_number (Número de Ordem) gerado no ato do agendamento.

4.2 Fluxo de Agendamento (End-to-End)

Login: Autenticação via Supabase Auth.

Onboarding: Se for o primeiro acesso, o sistema exige o preenchimento dos dados do Perfil.

Navegação: O militar visualiza o calendário. Apenas turnos com count < 21 e dentro da janela sazonal são habilitados.

Agendamento (RPC): O clique no botão dispara uma Function SQL (RPC) que:

Faz o lock da linha do turno (FOR UPDATE).

Verifica novamente se há vaga.

Insere o agendamento e gera o Número de Ordem (Ex: 2026-1-0012).

Confirmação: O sistema exibe o comprovante digital com os dados do agendamento e orientações.

5. Histórias de Usuário e Requisitos

5.1 Histórias de Usuário

US01: "Eu, como militar, quero escolher meu turno pelo celular para que eu possa planejar minha escala de serviço."

US02: "Eu, como coordenador, quero imprimir a lista de chamada ordenada para coletar as assinaturas no pátio de provas."

US03: "Eu, como sistema, devo impedir que o 22º militar agende no mesmo turno para manter a segurança da prova."

5.2 Requisitos Funcionais (RF)

RF01: Autenticação via e-mail e senha institucional.

RF02: Bloqueio de agendamento duplicado por semestre.

RF03: Geração de PDF do comprovante e da lista de chamada.

RF04: Dashboard de metas para a coordenação (Total de agendados vs. 380).

5.3 Requisitos Não Funcionais (RNF)

RNF01: Tempo de resposta do agendamento < 1.5 segundos.

RNF02: Site 100% responsivo para uso em dispositivos móveis Android/iOS.

RNF03: Isolamento de dados entre militares (Um não vê o nome do outro).

RNF04: Garantia de atomicidade (Sem agendamentos acima do limite).

6. Jornada do Usuário

Entrada: Login seguro.

Verificação: Sistema checa se o agendamento já foi feito no semestre atual.

Ação: Seleção de data disponível no grid responsivo.

Feedback: Notificação de sucesso com o Número de Ordem único.

Saída: Visualização do resumo e logout.

Plano de Projeto Detalhado: TACF-Digital (HACO)

Este documento define a estratégia de construção, as regras de negócio e a arquitetura técnica para o sistema de agendamento do Teste de Avaliação de Condicionamento Físico do Hospital da Aeronáutica de Canoas.

1. Contexto Estratégico

1.1 Justificativa e Problema

A gestão do TACF no HACO enfrenta desafios críticos:

Inconsistência de Dados: Agendamentos manuais permitem extrapolar o limite de 21 militares por turno.

Insegurança Jurídica (LGPD): Dados sensíveis de militares ficam expostos em listas físicas ou planilhas compartilhadas.

Logística de Pátio: A dificuldade em gerar listas de chamada ordenadas atrasa o início das avaliações físicas.

1.2 Objetivos Estratégicos

Controle de Quórum: Garantir janelas de 8 a 21 militares por turno de forma automatizada.

Eficiência de Escala: Gerenciar o fluxo para atingir a meta de 380 militares avaliados por semestre.

Modernização: Implementar uma interface web responsiva que dispense instalação (sem PWA), acessível de qualquer dispositivo.

1.3 Patrocinadores e Stakeholders

Patrocinador Principal: Comando do HACO.

Gestor do Projeto: Seção de Preparo Físico / Coordenação do TACF.

Interessados: Oficiais de Pessoal, Aplicadores de Teste e Militares da Guarnição.

2. Análise de Atores e Acesso

2.1 Perfis de Usuário

Militar (Avaliado): Acesso limitado aos seus próprios dados e agendamento.

3. Requisitos do Sistema

RF01 (Autenticação): Login via Supabase Auth com validação de perfil institucional.

RF02 (Onboarding): Preenchimento obrigatório de Posto/Graduação, Nome, Gênero e SARAM antes do primeiro agendamento.

RF03 (Geração de Ordem): O Número de Ordem deve ser gerado apenas no momento da confirmação final do agendamento (dentro da transação atômica).

RF04 (Relatórios): O sistema deve permitir que o Coordenador gere e imprima relatórios em formato A4 (PDF).

RF05 (Sazonalidade): Bloqueio de agendamento automático fora das janelas (FEV-MAI e SET-NOV).

3.2 Requisitos Não Funcionais (RNF)

RNF01 (Responsividade): O site deve ser 100% responsivo (Mobile-First). Não será utilizado PWA.

RNF02 (Stack): Construção em React 18, Vite 5, Yarn e TypeScript Estrito.

RNF03 (Estilização): Tailwind CSS v4 para garantir leveza e design institucional.

RNF04 (Atomicidade): Garantia de que o 22º agendamento seja rejeitado por bloqueio de linha no banco de dados (FOR UPDATE).

4. Histórias de Usuário (User Stories)

Militar: "Como militar, quero agendar meu teste pelo telemóvel no autocarro, recebendo meu número de ordem imediatamente após a reserva."

Coordenador: "Como coordenador, quero imprimir a lista de chamada ordenada pelo número de ordem para que eu possa organizar os pelotões no pátio."

Sistema: "Como sistema, devo gerar o número de ordem YYYY-S-XXXX apenas se a vaga for confirmada com sucesso."

5. Construção e Arquitetura Técnica

5.1 Relacionamento de Entidades (ER)

Tabela profiles: UUID (Auth), Nome, Posto, SARAM, Gênero, Role.

Tabela shifts: ID, Data, Período, Max_Capacity (default 21), Aplicadores (Array).

Tabela schedules: ID, Profile_ID, Shift_ID, Order_Number (Unique), Semestre, Ano.

5.2 Fluxo de Agendamento Detalhado

Login: O militar entra com suas credenciais.

Verificação de Perfil: Se o perfil estiver incompleto, redireciona para o formulário de Onboarding.

Seleção de Turno: O sistema exibe apenas datas com vagas (calculadas em tempo real).

Confirmar Agendamento:

O usuário clica em "Confirmar".

O sistema chama uma RPC (Function no Postgres).

A função bloqueia a linha do turno, verifica o quórum atual.

Se count < 21, realiza o INSERT.

Geração do Número de Ordem: No momento do INSERT, o banco concatena o Ano + Semestre + Sequencial daquela data.

Finalização: O frontend exibe a tela de sucesso com o Número de Ordem gerado.

6. Módulo de Relatórios Administrativos

O Coordenador terá acesso a uma área de "Relatórios e Impressão" com as seguintes opções:

Lista de Chamada Oficial: PDF formatado em A4, contendo Posto, Nome, SARAM e campo para assinatura, ordenado pelo Número de Ordem.

Relação de Aplicadores: Lista de turnos e seus respectivos oficiais/graduados responsáveis.

Relatório de Adesão: Estatística por Organização Militar (OM) ou Subdivisão, mostrando quem ainda não agendou.

Comprovantes em Lote: Impressão de todos os comprovantes de um turno específico para fins de arquivo físico.

Detalhamento de Fluxo: Autenticação e Onboarding

Este documento descreve o processo de entrada do usuário no sistema TACF-Digital, desde o primeiro acesso até a liberação para o agendamento.

1. Visão Geral do Fluxo de Entrada

O sistema utiliza um fluxo de Redirecionamento Inteligente. Ao invés de uma inscrição longa no início, dividimos o processo em duas etapas: Autenticação Rápida e Onboarding Institucional.

1.1 Etapa 1: Autenticação (Login)

Método: E-mail e Senha (via Supabase Auth).

Interface: Tela limpa, responsiva, com foco total no formulário de entrada.

Segurança: Uso de JWT (JSON Web Tokens) persistidos com segurança no navegador para evitar logins repetitivos.

1.2 Etapa 2: Verificação de Perfil (Inscrição)

Após o login, o sistema realiza uma consulta imediata à tabela profiles.

Cenário A (Perfil Incompleto): O usuário é bloqueado em uma tela de "Completar Perfil". Ele não visualiza o calendário de vagas enquanto não fornecer os dados obrigatórios.

Cenário B (Perfil Completo): O usuário é direcionado para o seu respectivo Dashboard (Militar ou Coordenador).

2. Processo de Inscrição de Perfil (Onboarding)

O formulário de inscrição é a base da integridade do sistema. Como não usamos PWA, este formulário é otimizado para preenchimento rápido em teclados virtuais de dispositivos móveis.

2.1 Campos Obrigatórios

Posto/Graduação: Seleção via Dropdown (ex: Tenente, Sargento, Cabo).

Nome Completo: Campo de texto (validação de nome e sobrenome).

SARAM (Identificador): Campo numérico obrigatório. Servirá como chave para busca de histórico.

Gênero: Necessário para aplicação correta dos índices do TACF no futuro.

Organização Militar (OM): Seleção pré-definida (ex: HACO, GSD-CO, etc.).

2.2 Regras de Validação (TypeScript)

SARAM: Deve conter exatamente 7 dígitos (exemplo).

Nome: Mínimo de 10 caracteres para evitar nomes incompletos.

Imutabilidade: Uma vez salvo, o SARAM e o Nome só podem ser alterados mediante solicitação ao Coordenador (evita fraudes de identidade).

3. Lógica Técnica de Login e Persistência

A construção segue o padrão Database-First com TypeScript:

// Exemplo de lógica de verificação de sessão
async function checkUserSession() {
  const { data: { user } } = await supabase.auth.getUser();
  
  if (user) {
    // Busca o perfil para saber se ele já se "inscreveu" (onboarding)
    const { data: profile } = await supabase
      .from('profiles')
      .select('*')
      .eq('id', user.id)
      .single();

    if (!profile || !profile.full_name) {
      return "/onboarding"; // Redireciona para completar inscrição
    }
    
    return profile.role === 'coordenador' ? "/admin" : "/dashboard";
  }
  return "/login";
}


4. Diferenciais de Segurança

4.1 Proteção contra "Brute Force"

O Supabase Auth lida com o bloqueio de tentativas sucessivas de login, protegendo as contas dos militares.

4.2 Row Level Security (RLS) no Login

Mesmo que um usuário mal-intencionado consiga logar, ele só conseguirá ler o perfil associado ao seu auth.uid(). Se ele tentar acessar o perfil de outro militar via console, o banco de dados retornará um erro silencioso (zero linhas).

5. Experiência do Coordenador

O Coordenador não passa pelo fluxo de agendamento comum. Seu login o leva diretamente ao painel de controle onde ele pode:

Visualizar quem realizou o login mas não completou o onboarding.

Imprimir relatórios de usuários pendentes.

Ajustar permissões de outros usuários para o nível de "Coordenador".

6. Layout Responsivo

Teclado: Atribuição correta de inputmode="numeric" para o SARAM para abrir o teclado numérico no celular.

Feedback: Mensagens claras de erro em caso de credenciais inválidas ou campos de perfil vazios, usando componentes de Toast (notificações rápidas).
